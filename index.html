What is Microfrontend?

The idea is to think about a website as a composition of features. On the basis of these feature, front-end app is decomposed into individual, semi-independent “microapps” which is loosely coupled. Each microapp can be managed by individual team.

Why we need Microfrontend?

Massive Restructuring

  • Current technology becomes deprecrated
  • It's not popular anymore
  • It's not suited for problem we are trying to solve
  • Redesigning UI for entire application
    • There are times when it becomes absolutely neccessary to rewrite whole codebase. Maybe current technology becomes deprecrated or maybe it's not popular anymore so it's not easy to find developers who can work on it or maybe current technology not suited for problem we trying to solve. But if we change technology of project having large codebase then it will take huge effort and time. As it happens when we switch from angularjs to Angular 2.

    Upgrading Framework

    It's not easy to upgrade a project when different teams working on same codebase. When I was ugrading angular 5 to angular 7 there were lots of issues coming because other teams pushing code parallely. So as I was giving new release then new bugs comes up. Also I had made changes in around 300 files so it was very difficult to review and test it.

    Refactoring code

                        
                        import {Component, OnInit} from '@angular/core';
                        import {Store} from '@ngrx/store';
                        import {FormGroup} from '@angular/forms';
                        import {LoginForm} from '../../../../app/core/models/LoginForm';
                        import {AppState} from '../../../../app/core/store/reducers/index';
                        import {LoginAction} from '../../../../app/core/store/actions/user/LoginAction';

                        @Component({
                          selector: 'gt-login-form',
                          templateUrl: 'gt-login-form.component.html',
                          styleUrls: ['gt-login-form.component.scss']
                        })
                        export class GtLoginFormComponent implements OnInit {
                          public model: LoginForm;

                          constructor(private store: Store) {
                          }

                          public ngOnInit() {
                            this.model = {
                              username: '', 
                              password: '',
                              valid: false
                            };
                          }

                          public onSubmit(loginForm: FormGroup) {
                            this.model.valid = loginForm.valid;
                            this.store.dispatch(new LoginAction(this.model));
                          }
                        }
    This is GtLoginFormComponent. No one is using it right now. But it is still part of linckia's build. There may be so many things which is not used but still be part of our codebase. It's not easy to clean codebase when you have large codebase and modules are too tightly coupled with each other. Our technical debt keep on increasing.

    Performance

    When we opens attendance page, we loads many components, service, css which it not needs to work. This is huge performance hindrance. We should load only those thing which it need. This problem keep on increasing as we add new module. There is no proper mechnasim for loading resourse.

    Testing

    With current architecture it's diffcult to do testing. Even for small change we have to test entire application.

    Code consistency, Coding standard

    When different team working across on same project then it's pretty hard to maintain contistency. For example in linckia in some place we have used store for state management and in some place we are using service. So code readablity decreased

    Resilence

    It may happen if someone has written bad code then it may effect whole project.

    Better Maintenance

    Instead of huge holistic codes, the code-base is distributed according to the feature and functionality and a specific team maintains them. This makes it easier to maintain the app.

    High Resilience

    In Micro frontend web apps are distributed in small chunks of services starting from the UI to database that works independently. So, even if one feature faces the issue, it doesn’t affect the rest of the web app. And this feature makes it possible to recover from the error faster.

    Highly Scalable

    Because of its modular structure micro frontend is highly scalable when you need to grow the business.

    Simple, decoupled codebases

    The source code for each individual micro frontend will by definition be much smaller than the source code of a single monolithic frontend. These smaller codebases tend to be simpler and easier for developers to work with.

    Flexibility in Using Technologies

    You do know by now that each micro frontend can be written using a different technology. This simplifies the selection of the most appropriate tech stack for the specific needs of your service.

    Improved Productivity and Speed

    The microservices architecture tackles the problem of productivity and speed by decomposing applications into manageable services that are faster to develop. Different teams can be working on different components simultaneously without having to wait for one team to finish a chunk of work before starting theirs.

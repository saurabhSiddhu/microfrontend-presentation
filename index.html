    Evolution of Web Development

    Before moving to micro front-end, let's go back from where we came
    In the beginning, we have monolithic system where everything is bundled up inside a single deployable unit. This monolithic used backend templating engine like freemarkup and jquery for ui. The system is highly coupled. As SPA become popular we split big monolithic into two parts front-end and backend. Now we have front-end team and backend team which working with more focussed view of app and code became more maintainable. But as application grows bigger and complex it became not feasable to build everything in single project so we divided backend into small services. In all these common is front-end remain monolithic. In recent time the frontend layer became more and more sophisticated. User expect more interactive experinces. We don't have mechanism for dealing with it. This is where Microfrontend comes in.

    Next Step Of Evolution

    The idea is to think about a website as a composition of features. On the basis of these feature, front-end app is decomposed into individual, semi-independent “microapps” which are loosely coupled. Each microapp can be managed by a independent team. Each team has a distinct area of business or mission it cares about and specialises in. A team is cross functional and develops its features end-to-end, from database to user interface.

    Why we need Microfrontend?

    Development Speed

    The Microfrontend architecture tackles the problem of productivity and speed by decomposing applications into manageable chunks that are faster to develop. Since teams are autonomous therefore different teams can be working on different components simultaneously without having to waiting for other team to finish work before starting theirs.

    Massive Restructuring

  • Current technology becomes deprecrated
  • It's not popular anymore
  • It's not suited for problem we are trying to solve
  • Redesigning UI for entire application
    • There are times when it becomes absolutely neccessary to rewrite whole codebase. Rewriting takes huge effort and time. As it happens when we switch from angularjs to Angular 2. But if we were using Microfrontend architecture then it would not that difficult to migrate. We can do it in piece by piece way

    Reduced Scope

    When we works in large projects, we tries to know all things related to what we working on. But chances are there may be some blury part which you not aware of. You do something and some other things break which you don't know. But working in Microfrontend reduces scope so we now able to know everything about project.

    Refactoring code

                        
                        import {Component, OnInit} from '@angular/core';
                        import {Store} from '@ngrx/store';
                        import {FormGroup} from '@angular/forms';
                        import {LoginForm} from '../../../../app/core/models/LoginForm';
                        import {AppState} from '../../../../app/core/store/reducers/index';
                        import {LoginAction} from '../../../../app/core/store/actions/user/LoginAction';

                        @Component({
                          selector: 'gt-login-form',
                          templateUrl: 'gt-login-form.component.html',
                          styleUrls: ['gt-login-form.component.scss']
                        })
                        export class GtLoginFormComponent implements OnInit {
                          public model: LoginForm;

                          constructor(private store: Store) {
                          }

                          public ngOnInit() {
                            this.model = {
                              username: '', 
                              password: '',
                              valid: false
                            };
                          }

                          public onSubmit(loginForm: FormGroup) {
                            this.model.valid = loginForm.valid;
                            this.store.dispatch(new LoginAction(this.model));
                          }
                        }
    This is GtLoginFormComponent. No one is using it right now. But it is still part of linckia's build. There may be so many other things which is not used but still be part of our codebase. It's not easy to clean codebase when you have large codebase and modules which are too tightly coupled with each other.

    Performance

    When we opens attendance page, we loads all components, service, css . This is huge performance hindrance. At first, we should only load those resources which are absolutely necessary. This problem keep on increasing as we add new module. There is no proper mechanism for loading resourse. But as Microfrontend architecture divide our codebase into chunks it would be easier for us to acheive it.

    Testing

    With current architecture it's diffcult to do testing. As our code have tightly coupled, Even for small change we have to test entire application.

    Code consistency, Coding standard

    When different team working across on same project then it's pretty hard to maintain contistency. For example in linckia in some place we have used store for state management and in some place we are using service. So code readablity keeps on decreasing with time. Howerver in Microfrontend, as teams are working on independent microapp , it would be easy to enforce coding standards across the app.

    Upgrading Framework

    For new features, better security, support and performance we have to keep on upgrading our framework. It's not easy to upgrade a project when different teams working on same codebase and we have huge codebase . When I was ugrading angular 5 to angular 7 there were lots of issues coming because other teams pushing code parallely. Also I had made changes in around 300 files so it was very difficult to review and test. Since in Microfrontend code divided into small chunks , it is easy to upgrade

    Resilence

    It may happen if someone has written bad code then it may effect whole project. But In Micro frontend web apps are distributed in small chunks of services starting from the UI to database that works independently. So, even if one feature faces the issue, it doesn’t affect the rest of the web app. And this also makes possible to recover from the error faster.

    Maintenance

    Instead of huge code, the code-base is distributed according to the feature and functionality and a specific team maintains them. This makes it easier to maintain the app.

    Simple, decoupled codebases

    The source code for each individual micro frontend will by definition be much smaller than the source code of a single monolithic frontend. These smaller codebases tend to be simpler and easier for developers to work with.

    Flexibility in Using Technologies

    You do know by now that each micro frontend can be written using a different technology. This simplifies the selection of the most appropriate tech stack for the specific needs of your service.
